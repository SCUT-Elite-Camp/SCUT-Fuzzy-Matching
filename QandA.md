## 1. 分簇的具体依据是什么？

**依据：200 维 MinHash 签名向量之间的余弦相似度。**

具体流程（见 [party_b/offline_prep.py](party_b/offline_prep.py) 和 [clustering/kmeans_cosine.py](clustering/kmeans_cosine.py)）：

1. **生成 200 维 MinHash 签名** — 对 B 侧每个姓名做 trigram（3 字符滑动窗口）分片 → 200 个哈希置换 → 取每个置换的最小哈希值，得到 `(n, 200)` 的签名矩阵。

2. **L2 归一化** — 对每个姓名向量做行 L2 归一化，使其落在单位超球面上。此时两向量点积 = 余弦相似度。

3. **StandardScaler 标准化** — 对归一化后的 200 维矩阵做 Z-score 标准化（减均值、除标准差），使各维度均值为 0、方差为 1。这确保聚类不会被某些数值偏大的维度主导。

4. **Cosine/Spherical K-Means** — 在标准化后的 200 维空间上运行余弦 K-Means（[kmeans_cosine.py:27-74](clustering/kmeans_cosine.py#L27-L74)）：
   - **分配**：每个样本分配给余弦相似度最高的质心（`argmax(cosine_similarity)`，而非欧氏距离最近的质心）。
   - **更新**：质心 = 簇内成员均值向量的 L2 归一化（使其重新落在单位球面上）。
   - **k 值**：默认 `k = ⌊√n⌋`（[params.py:14](config/params.py#L14)），n 为 B 侧姓名总数。

**直观理解**：MinHash 将姓名编码为向量，相似的姓名（如 "John" 和 "Jonh"）会产生相似的 MinHash 签名。通过余弦 K-Means 将签名空间中的"近邻"划入同一簇，后续只需在最有希望的那一簇中做精细匹配，避免全库遍历。

## 2. 200 维和 50 维向量分别代表什么？是依据什么得到的？分别有什么用？

### 它们是什么？

两者都是**同一个姓名经过 MinHash 编码后的签名向量**，只是取了不同数量的哈希置换（permutations）：

| | 200 维 | 50 维 |
|---|---|---|
| **参数名** | `NUM_PERMUTATIONS_CLUSTER = 200` | `NUM_PERMUTATIONS_MATCH = 50` |
| **含义** | 使用 200 个哈希函数对姓名的 trigram 集合做 MinHash | 使用前 50 个哈希函数（200 维的前缀） |

### 生成方式

```text
姓名 "john" → 清洗 → trigrams: {"joh", "ohn"} → 哈希 → 200 个置换取最小 → (200,) 向量
                                                                         ↓
                                                              取前 50 维 → (50,) 向量
```

关键约束（[encoder.py:54-76](minhash/encoder.py#L54-L76)）：50 维向量必须是 200 维向量的**前缀**（前 50 维），使用**同一组哈希种子**（`HASH_SEED = 42`）。不能重新生成不一致的签名。

### 直观理解

你可以把 MinHash 想象成给每个姓名拍了一张"指纹照片"：

- **200 维** = 高分辨率照片（200 个像素点）。信息量大、区分度高，适合用来做**粗粒度的分组**——把几万个姓名分到 k 个簇中。就好比先用模糊的全景照片判断"这个人大致在这一片区域"。

- **50 维** = 低分辨率缩略图（只取前 50 个像素点）。信息紧凑、加密计算开销小，适合在簇内做**精匹配**——将查询姓名与簇内每个候选姓名逐一计算加密相似度。就好比靠近后用人脸特征做精确比对。

### 各自的用途

| 向量 | 用途 | 出现在哪一步 |
|---|---|---|
| **200 维** | 聚类（划分 B 侧姓名库） | Step 1 |
| **200 维**（标准化后） | 第一轮质心匹配（判断查询属于哪个簇） | Step 2–3 |
| **50 维**（仅 L2 归一化） | 第二轮列式匹配（簇内逐姓名计算余弦相似度） | Step 6–7 |

**为什么不统一用 50 维或 200 维？**
- 200 维区分度更好，用于聚类和选簇更准确。
- 50 维向量在 CKKS 加密下同态运算更快（每次点积计算量更小），且 B 侧簇内候选姓名数通常不大（几十到几百的量级），50 维已足够分辨。

## 3. 整个项目流程加解密了几次？用的是同一组公私钥吗？

### 加解密总览

整个协议涉及 **3 次加密、2 次解密**，使用**同一组公私钥**（由 A 侧在 Step 2 创建的一个 CKKS 上下文生成）。

```
A 侧（Party A）                        B 侧（Party B）
═══════════════                        ═══════════════

Step 2: 创建 CKKS 上下文（含公私钥）
  ├─ 加密 query_200 → 发给 B  ──────→  (加密 #1)
  └─ 加密 query_50 → 留在 A 本地     (加密 #2)

Step 3:                             用公开上下文做同态计算
                                       dot_ct_pt(enc_query_200, 每个质心)
                                       结果 → 发回 A

Step 4: 解密质心相似度               (解密 #1)
  → 选最高分的簇 → one-hot 向量
  → 加密 one-hot → 发给 B  ────────→  (加密 #3)

Step 6-8:                           用公开上下文做同态计算
                                       matmul_ct_pt(enc_selector, 列矩阵)
                                       dot_ct_ct(enc_query_50, enc_selected_name)
                                       加减随机掩码

Step 9: 逐列解密分数判断正负          (解密 #2)
  → 输出 catch (是否匹配)
```

### 总结

| 步骤 | 操作 | 对象 | 执行方 |
|---|---|---|---|
| Step 2 | **加密 #1** | `query_200_std`（200 维） | A 侧 |
| Step 2 | **加密 #2** | `query_50_norm`（50 维） | A 侧 |
| Step 5 | **加密 #3** | `selector`（k 维 one-hot） | A 侧 |
| Step 4 | **解密 #1** | 质心相似度（k 个标量） | A 侧 |
| Step 9 | **解密 #2** | 列匹配分数（逐列标量） | A 侧 |

### 密钥关系

**是同一组公私钥。** A 侧在 `create_ckks_context()` 中创建上下文（[context.py:11-33](ckks/context.py#L11-L33)），其中自动包含新生成的公钥和私钥。A 将「公开上下文」通过 `serialize_public_context()` 移除私钥后发给 B（[keys.py:10-33](ckks/keys.py#L10-L33)），B 只能用公钥做同态运算，永远无法解密。A 保留含私钥的完整上下文（`PartyALocalState.secret_context`），所有解密都在 A 侧完成。

## 4. "簇选择向量 × 聚类矩阵"与"加密查询向量・加密姓名向量"分别怎么算？为什么这么算？

### 第一式：簇选择向量 × 聚类矩阵 → 加密姓名向量

**对应代码**：[online_responder.py:114-116](party_b/online_responder.py#L114-L116)

```python
encrypted_selected_name_j = matmul_ct_pt(encrypted_selector, masked_column_j)
```

**实现**：[operations.py:34-51](ckks/operations.py#L34-L51) — 密文向量 × 明文矩阵

- `encrypted_selector`：形状 `(k,)` 的密文向量（one-hot，加密后）
- `masked_column_j`：形状 `(k, 50)` 的明文矩阵
- 结果：形状 `(50,)` 的密文向量

**计算方式**（密文-明文矩阵乘法 `enc_vec.matmul(plain_matrix)`）：

$$E(\text{selected\_name}_j) = \sum_{i=0}^{k-1} E(\text{selector}[i]) \cdot \text{column}_j[i, :]$$

即 **selector 向量与矩阵的叉乘（向量-矩阵乘法）**。selector 是行向量，column_j 是 `(k, 50)` 矩阵，结果是 50 维向量。

**直观理解**：selector 是"选第几簇"的遥控器（one-hot 加密），`column_j` 是"每簇第 j 列名字的 50 维向量"排成的表格。叉乘的效果是：把遥控器指向的那一行（那一个名字向量）挑出来。由于 selector 是加密的，B 侧执行了这个"挑选"操作却不知道挑了哪一行——就像闭着眼睛按遥控器，信号发出去了，但你不知道自己按的是哪个频道。

**为什么用叉乘（matmul）而不是直接索引？**
因为 selector 是加密的，B 无法知道 `argmax(selector)` 是几，所以不能直接用 `column_j[argmax, :]` 取行。矩阵乘法在密文下实现了等价的"加权求和选择"——one-hot 中只有一个位置为 1，乘加后正好拿出那一行，而其余行被 0 乘掉。

---

### 第二式：加密查询向量・加密姓名向量 → 加密相似度值

**对应代码**：[online_responder.py:117-119](party_b/online_responder.py#L117-L119)

```python
encrypted_cos_score_j = dot_ct_ct(encrypted_query_50, encrypted_selected_name_j)
```

**实现**：[operations.py:54-66](ckks/operations.py#L54-L66) — 密文-密文点积

- `encrypted_query_50`：形状 `(50,)` 的密文向量（A 的查询）
- `encrypted_selected_name_j`：形状 `(50,)` 的密文向量（上一步选出的候选姓名）
- 结果：密文标量

**计算方式**（`enc_left.dot(enc_right)`——TenSEAL 内置的槽位逐元素乘加）：

$$E(\text{cos\_score}_j) = \sum_{i=0}^{49} E(\text{query}[i]) \cdot E(\text{name}_j[i])$$

即 **两个等长向量的点积（内积）**。

**直观理解**：两个加密向量"在黑盒里"逐元素相乘再求和。由于双方向量都事先做了 L2 归一化（长度为 1），点积的结果天然就是余弦相似度。这一步是加密模糊匹配的核心——B 在完全不知道两个姓名内容的情况下，计算出了它们的相似度。

**为什么用点积而不是其他运算？**
因为**L2 归一化 + 点积 = 余弦相似度**。两个单位向量的点积就是它们夹角的余弦值。如果两个姓名完全相同，点积 ≈ 1；如果完全不同，点积 ≈ 0。这是衡量向量相似度的最自然方式，而且点积在 CKKS 下可以直接用同态乘加高效实现。

---

### 两式的关系与整体协议设计

```
第二轮的完整数据流（B 侧视角）：

encrypted_selector(k,)  ×  column_j(k,50)   →  encrypted_selected_name_j(50,)
        ↑                      ↑                        ↑
   "选哪个簇"           "每簇第j列的名字"          "被选中的名字向量"
  (加密的遥控器)          (明文表格)                (加密的结果)

                                          encrypted_query_50(50,)
                                                 ↑
                                         "A 的查询姓名"
                                                 ↓
encrypted_query_50(50,) ・ encrypted_selected_name_j(50,)  →  encrypted_cos_score_j(标量)
                                                                      ↓
                                                          - τ (阈值) × r_j (随机掩码)
                                                                      ↓
                                                          逐列发给 A 解密判断正负
```

**为什么这样设计两层？**
1. **第一步（叉乘/选择）** 把"k 个簇"的搜索空间缩到"1 个簇"，而且 B 不知道选了哪个。
2. **第二步（点积/匹配）** 在选中的簇内，逐列（逐姓名）计算加密相似度。因为是逐列进行，A 可以在发现匹配时提前停止（early stop），节省后续计算。
3. **B 全程不知道**：A 查询了谁、A 选了哪个簇、A 和哪个名字匹配了。B 只知道自己在"帮忙算"。A 最终只知道"是否匹配"（以及 debug 信息保留在本地的第一个命中列位置）。

## 5. 只考虑了姓名匹配，为什么不用身份证号这种唯一标识？

### 身份证号匹配是另一个问题

身份证号（或 SSN、护照号）是**精确匹配**（exact match）场景，不涉及模糊匹配。两者适用于不同的需求：

| | 姓名模糊匹配（本项目） | 身份证号精确匹配 |
|---|---|---|
| **输入性质** | 含拼写错误、变体、音译差异 | 固定格式，无歧义 |
| **需要的技术** | MinHash + 余弦相似度 + 阈值 | 哈希比对 / 加密等值判断（如 PSI） |
| **典型错误** | "John" vs "Jonh"、全名缩写 | 输错一位数字 |
| **匹配算法** | 近似匹配（fuzzy） | 精确匹配（exact） |

### 为什么本项目聚焦姓名

从项目使用的数据集 `ncvr_10k`（NCVR = North Carolina Voter Registration）可以看出，这是一个**选民登记去重场景**：不同选区的选民注册记录中，同一个人可能以略微不同的姓名出现（如 "Robert Smith" vs "Bob Smith" vs "Robt Smith"），但身份证号（SSN）因隐私法规不能直接共享或跨库比对。

身份证号虽然能唯一标识一个人，但：
1. **隐私敏感度更高** — 身份证号比姓名更需要加密保护，但其加密匹配方案是"等值判断"而非"相似度计算"，与本项目的同态加密相似度框架不兼容。
2. **并非所有场景都有身份证号** — 选票登记、历史档案、跨机构协作等场景通常只有姓名。
3. **现实中的配合使用** — 在完善的系统中，姓名模糊匹配是第一道关卡（召回候选集），再用身份证号做精确确认（排序或过滤）。本项目的基线只覆盖模糊匹配这一环。

## 6. 无法解决同名的问题且仅限英文姓名，后期打算怎么优化？

### 同名问题

**同名问题**（如两个不同的人是 "John Smith"）在当前基线中确实无法区分——MinHash 签名完全相同，余弦相似度 = 1，协议必然返回 `catch=True`。

### 后期优化方向

参考项目规范文档中的扩展预留和同态加密领域的通用做法：

**方向一：引入多维特征（多字段联合匹配）**

当前基线只对姓名这一个字段做模糊匹配。可以在同一个 CKKS 框架下扩展为多字段联合向量：

```
当前：姓名_embedding → (200 / 50 维)
扩展：姓名_embedding + 出生日期 + 地址 + 邮编 → 拼接为 (200+ 维)
```

由于 CKKS 支持槽位打包（SIMD），多字段编码后仍可在一次同态点积中完成匹配。同名 + 同生日 + 同地址的两个人概率极低，大大降低了同名误报。

**方向二：批次查询 + 排序（m > 1）**

当前基线是单查询 `m=1`，规范中明确提到批处理是后续扩展方向（见 [五人分工接口规范_agent版.md §1.5](docs/五人分工接口规范_agent版.md)）。多查询场景下，A 可以同时发出多条姓名变体，对返回的匹配分数排序取 top-k，而不是仅靠单个查询的 catch 信号。

**方向三：二次精确过滤**

第一轮模糊匹配（本项目）返回加密候选集 → 第二轮用额外的精确字段（如加密的身份证号等值比对）确认。本项目作为基线只实现了第一轮。

### 仅限英文姓名的优化

当前 `clean_name()` 函数（[text_cleaner.py:7-16](preprocessing/text_cleaner.py#L7-L16)）的处理流程：

```python
name = name.lower().strip()
name = re.sub(r"[^a-z\s]", "", name)  # 只保留 a-z 和空格
```

这会过滤掉所有中文、重音字符（é、ü）、连字符（Jean-Pierre）等。

**优化方向：**

| 语言 | 当前问题 | 改进方式 |
|---|---|---|
| **中文** | 汉字被直接删除 | 改用单字/分词 shingling，或先转为拼音再做 trigram |
| **带重音的欧洲语言** | "José" → "jos"（丢失 é） | NFKC 归一化已有（`unicodedata.normalize`），但正则 `[^a-z\s]` 会删除。改为先做 Unicode 音调剥离（如 `é→e`、`ü→u`），再匹配 |
| **连字符姓名** | "Jean-Pierre" → "jeanpierre" | 当前正则已删除 `-`。可改为保留连字符或空格替代，使 "Jean-Pierre" 和 "Jean Pierre" 可匹配 |
| **阿拉伯语/西里尔字母** | 全部被过滤 | 扩展正则以保留对应 Unicode 范围，或先做罗马化转写 |

**具体的代码改动预案**（已在规范中提及为后续工作）：

```python
# 当前
name = re.sub(r"[^a-z\s]", "", name)

# 扩展方向：保留更多 Unicode 字母
name = re.sub(r"[^\w\s]", "", name)          # 保留所有 Unicode 字母
# 或分语言策略：中文先转拼音，西文做音调剥离
```

## 7. 为什么用 K-Means 聚类算法，不用 DBSCAN 或别的算法？

### K-Means 在此协议中的天然适配

K-Means 被选用的原因根植于**两轮协议的结构需求**，而不是单纯"哪个聚类效果好"：

| 协议需求 | K-Means 如何满足 | DBSCAN / 其他算法的问题 |
|---|---|---|
| **固定数量的簇（k）** | 预先指定 k，输出恰好 k 个簇 | DBSCAN 簇数量不固定，取决于 eps 参数和数据密度 |
| **每个簇有质心（centroid）** | 每个簇有明确的中心向量（200 维），可直接用于第一轮质心匹配 | DBSCAN 没有质心概念；层次聚类需额外计算簇代表 |
| **规整的簇矩阵（cluster_matrix）** | 可以构建规整的 `(k, max_size, 50)` 三维矩阵，用零向量补齐 | DBSCAN 产生噪声点（不属于任何簇）和极不均匀的簇大小，矩阵会非常稀疏 |
| **每个样本都属于一个簇** | 所有样本都分配到某个簇，无"离群点" | DBSCAN 将部分样本标记为噪声（-1），这些样本永远不会被匹配到 |

### 两轮协议对聚类算法的强约束

回顾协议的两轮设计：

```
Round 1: encrypted_query_200 ・ centroid_i → 选出最匹配的簇
Round 2: 在选中的簇内逐列做 encrypted_query_50 ・ name_j
```

这个设计的核心前提是：
1. **B 能提供 k 个质心** → A 在第一轮用加密查询和每个质心算相似度，选最高分的簇。
2. **B 能按簇组织所有名字为规整的三维矩阵** → 第二轮能逐列扫描。

DBSCAN 破坏了这两个前提——没有质心、没有规整的簇结构。

### 其他算法的对比

| 算法 | 判决 |
|---|---|
| **DBSCAN** | ❌ 无质心、噪声点无法参与匹配、簇大小极不均匀 |
| **层次聚类（Hierarchical）** | ❌ 无明确质心，需额外抽取；树结构不适合"选一个簇再扫描"的协议 |
| **谱聚类（Spectral）** | ❌ 需要全库相似度矩阵（n×n），10k 规模即爆炸；无直接质心 |
| **高斯混合模型（GMM）** | 有质心（均值），但带协方差，第一轮相似度计算需改用马氏距离。复杂度高于 K-Means 而本场景精度提升有限 |
| **K-Means（欧氏）** | 接近但不精确匹配余弦语义。本项目改造为 Cosine K-Means |
| **Cosine K-Means（本项目）** | ✅ 质心清晰、k 固定、簇分配均匀、矩阵规整、余弦语义与 MinHash+L2 归一化完美契合 |

### 工程上的考量

`K_CLUSTERS_FUNC = sqrt(n)` 让 k 随数据库规模自动缩放。对于 10k 条记录，k ≈ 100，每个簇平均 100 条名字，第二轮匹配的计算量与通信量可控。如果换成 DBSCAN，可能出现一个超大的簇（上千条）和一个超小簇（几条），协议性能不可预测、难以落地。

## 8. 为什么采用余弦相似度，不采用欧式距离或其他相似度度量？

### 余弦相似度与 L2 归一化的天然配合

整个流水线中有这个关键步骤（[offline_prep.py:36-37](party_b/offline_prep.py#L36-L37)）：

```python
normalized_200 = l2_normalize(signatures_200)
normalized_50 = l2_normalize(signatures_50)
```

L2 归一化将每个向量映射到单位超球面上（向量长度为 1）。**在单位向量上，点积 = 余弦相似度。** 这意味着：

$$E(\text{query}) \cdot E(\text{name}_j) = \cos(\theta) = \text{余弦相似度}$$

一步同态点积就完成了相似度计算，不需要额外的分母或开方。

### 各度量在 CKKS 同态加密下的可计算性

| 相似度度量 | 公式 | CKKS 下的计算 | 可行性 |
|---|---|---|---|
| **余弦相似度（L2 归一化后）** | $\vec{a} \cdot \vec{b}$ | 一次 dot product | ✅ 天然支持 |
| **余弦相似度（原始）** | $\frac{\vec{a}\cdot\vec{b}}{|\vec{a}||\vec{b}|}$ | 需要同态除法和开方（求模长） | ❌ CKKS 不支持直接的同态除法/开方 |
| **欧氏距离** | $\sqrt{\sum(a_i - b_i)^2}$ | 展开为 $\sum a_i^2 + \sum b_i^2 - 2\sum a_i b_i$，仍需要同态开方 | ❌ 开方在 CKKS 中极其昂贵 |
| **Jaccard 相似度** | $\frac{|\text{交集}|}{|\text{并集}|}$ | 分子分母均需在加密状态下统计和除法 | ❌ 集合运算无法直接映射到向量同态 |
| **编辑距离（Levenshtein）** | 动态规划 | 条件分支和循环无法在同态加密下执行 | ❌ 根本上不兼容 |

### 直观理解

MinHash 本质上是在近似 Jaccard 相似度，但 Jaccard 无法在加密下高效计算。MinHash 将集合相似度转化为向量相似度后，搭配 L2 归一化，**余弦相似度成了在 CKKS 下唯一可直接用一个 dot product 即可准确计算的度量**。

欧氏距离虽然在明文下等效（对归一化向量，欧氏距离和余弦相似度是单调关系），但在密文下必须处理开方操作——CKKS 的 `sqrt` 需要用多项式近似或牛顿迭代，引入额外乘法深度和噪声，且消耗更多同态运算层数（level budget）。

### 余弦相似度对 MinHash 签名的语义契合

MinHash 签名的每个维度是"该哈希置换下的最小哈希值"。两个姓名共享的 trigram 越多，对应维度的最小值越可能相同。**在签名空间中，方向（角度）比距离（绝对值）更能表达相似性**——两个向量指向同一方向说明它们在各维度上的值分布相似，即底层 trigram 集合相似。余弦相似度捕捉的正是这种"方向一致性"。

## 9. 该方案的亮点在什么地方？

### 亮点一：全流程隐私保护的无交集计算

传统的"姓名比对"需要一方把数据发给另一方，或双方把数据交给第三方。本方案实现了 **三盲**（Triple-Blind）：

```
A 不知道 B 的数据库内容（只收到加密分数，最后只知道 "catch=True/False"）
B 不知道 A 查询的是谁（全部操作在密文下完成）
B 不知道 A 选中的是哪个簇（encrypted_selector 对 B 不可读）
B 不知道 A 最终和哪个名字匹配（分数带随机掩码，只保留正负号）
```

这在选民登记、反洗钱（AML）名单筛查、跨机构黑名单共享等场景有直接实用价值——各方在法律法规下不能交换原始数据，但需要知道"这个人是否在名单中"。

### 亮点二：两轮渐进式协议 — 用聚类把"全库扫描"变成"单簇扫描"

朴素方案需要对 B 侧全部 n 条姓名逐条做同态点积，O(n) 的同态运算量和通信量。

本方案用 Cosine K-Means 聚类把 n 条名字分成 k 个簇：

```
Round 1: k 次密文-明文点积（k ≈ √n）  → 选出目标簇
Round 2: max_size 次密文-密文点积      → 只扫描一个簇
```

| | 朴素方案 | 本项目 |
|---|---|---|
| **密文-明文点积** | 0 | k 次（轻量） |
| **密文-密文点积** | n 次 | max_size ≈ n/k 次（重量） |
| **通信轮数** | 1 轮 | 2 轮 |
| **提前停止** | ❌ | ✅ 发现正分立即停止 |

对于 n=10000, k=100，max_size≈100。朴素方案需 10000 次密文-密文点积（CT-CT dot 比 CT-PT dot 昂贵得多）；本方案只需 100 次 CT-PT + 最多 100 次 CT-CT，且早期命中可提前终止。**以多一轮通信为代价，大幅降低了昂贵同态运算的绝对数量。**

### 亮点三：随机掩码机制保护相似度绝对值

B 在产生加密相似度分数时（[online_responder.py:111-112](party_b/online_responder.py#L111-L112)），对每一列施加随机掩码：

```python
mask = _sample_positive_mask()                  # 随机正数 r_j ∈ [1, 10]
encrypted_score_j = (cos_score_j - τ) * r_j     # 同态乘法
```

这一操作有两个效果：
1. **A 只能判断正负，不知道真实相似度**。如果 `cos_score > τ`，结果为正；否则为负。但 A 无法从 `(cos-τ)*r` 反推 cos 的具体值。
2. **每列不同的随机掩码防止了列间的关联分析**。如果不用随机掩码，A 可以对比各列分数推断出第二相似的姓名，从而逐步重建数据库内容。

这是一个**用极低的同态计算代价（一次明文乘法、一次明文加法）换取强隐私保护**的设计。

### 亮点四：early stop 机制 — 隐私与效率不冲突

早期的隐私匹配方案往往需要在"全量计算"和"泄露搜索模式"之间二选一。本方案的 early stop（[online_querier.py:131-140](party_a/online_querier.py#L131-L140)）：

```python
if plain_score > eps and first_positive_column is None:
    first_positive_column = column_index
    if early_stop:
        return MatchResult(catch=True), ...   # 立即停止，不再消费后续列
```

因为是逐列流式返回（`yield`），A 发现正分后可立即停止消费 B 的返回流，B 无法区分"A 在第一列就找到了"还是"A 扫描完了所有列都没找到"——这个信息被掩藏在流式交互中。

### 亮点五：Cosine K-Means（而非标准欧氏 K-Means）

项目中实现的是 Cosine K-Means（[kmeans_cosine.py](clustering/kmeans_cosine.py)），它把余弦相似度融入聚类本身。这比先用欧氏 K-Means 聚类再在簇内用余弦匹配更一致——聚类时的"近邻"定义与匹配时的"相似"定义完全相同，都是余弦相似度。减少了"欧氏聚类下属于同一个簇、但余弦相似度低"的结构性误差。

### 亮点六：模块化五人分工与安全边界强制

从 [文档](docs/五人分工接口规范_人类版.md) 和代码设计可以看出，A/B 两侧的安全边界通过**类型系统 + 数据结构 + 代码组织**强制执行：

- `FirstRoundRequest` 只含 `encrypted_query_200` 和公开上下文——`encrypted_query_50` 在物理上无法泄露到第一轮。
- `PartyALocalState`（含私钥）只存在于 A 侧文件——B 侧函数不接收 `secret_context`。
- `selected_cluster`（明文）只存在于 `ClusterSelectionDebug`——不进入 `SecondRoundRequest`。

这些边界不是靠口头约定，而是靠代码设计保证的。测试（[tests/test_security_boundary.py](tests/test_security_boundary.py)）可以自动化验证这些边界未被破坏。

### 亮点七：MinHash 前缀一致性 — EL=50 是 EL=200 的前 50 维

这一设计（[encoder.py:38-51](minhash/encoder.py#L38-L51)）避免了两次独立的 MinHash 生成带来的不一致——50 维和 200 维共享同一组哈希函数种子，保证 50 维是 200 维的真子集，聚类空间和匹配空间的语义完全对齐。

---

### 总结：核心权衡与本方案的平衡点

本方案的本质是在 **隐私强度、计算效率、匹配精度** 三者之间找到了工程上可行的平衡点：

- **隐私**：全同态加密 + 随机掩码 + 早期停止，A 和 B 互不暴露数据。
- **效率**：两轮聚类协议 + early stop，将 O(n) 的同态运算推进到 O(√n) + O(n/k)。
- **精度**：Cosine K-Means + 余弦相似度保证了聚类语义和匹配语义的一致性。

它不是理论上"最优"的解法（例如完全同态的 SMID 批处理方案可能更高效，或在非加密场景下 MinHash LSH 可以更快），但在**可落地、可理解、可验证**三个维度上是一个出色的基线系统。
