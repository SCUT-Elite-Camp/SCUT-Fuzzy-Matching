# 一、项目整体目标

系统目标：

在不泄露查询姓名与数据库内容的前提下，实现：

* 模糊姓名匹配（Fuzzy Name Matching）
* 隐私保护（Privacy Preserving）
* 高召回率（Recall ≥ 96%）
* 高精度（Precision ≈ 100%）
* 低通信开销
* 可扩展到 10k / 100k / 1M 数据规模

核心技术路线：

```text
MinHash
+ K-Means Clustering
+ CKKS Fully Homomorphic Encryption
```

系统角色：

| 角色                 | 说明              |
| ------------------ | --------------- |
| Party A / Client A | 查询方，持有待查询姓名     |
| Party B / Server B | 数据持有方，持有私有姓名数据库 |

最终：

* A 不泄露查询内容
* B 不泄露数据库
* A 仅得到“是否存在近似匹配”

---

# 二、整体协议流程概览

整体协议可以理解为：

```text
Server B:
离线构建索引
→ 聚类
→ 构建列式矩阵

Client A:
加密查询

Server B:
密文计算

Client A:
解密并判断是否匹配
```

整体分为：

| 阶段      | 内容               |
| ------- | ---------------- |
| PHASE 0 | B 离线预处理          |
| PHASE 1 | A 本地查询预处理        |
| PHASE 2 | 第一轮通信：质心匹配       |
| PHASE 3 | 第二轮通信：发送 one-hot |
| PHASE 4 | 列式模糊匹配           |
| PHASE 5 | A 最终判断           |

# 1. Clustering（Server B 离线预处理）

对应流程图左下角：

```text
1. Clustering
```

这是整个系统的离线建库阶段。

---

## 1.1 MinHash 编码

Server B 对数据库中所有姓名：

```text
姓名
→ trigram shingling
→ hash
→ permutation
→ MinHash signature
```

得到：

| 编码     |  维度 | 用途        |
| ------ | --: | --------- |
| EL=200 | 200 | 聚类 / 质心匹配 |
| EL=50  |  50 | 最终名字匹配    |

---

## 1.2 L2 Normalization + StandardScaler

对 EL=200：

```text
L2 Normalize
→ StandardScaler
```

原因：

* 后续使用余弦相似度
* K-Means 需要统一向量空间
* A/B 必须共享同一个标准化空间

B 会共享：

```text
scaler.mean_
scaler.scale_
```

给 A。

---

## 1.3 K-Means 聚类

在：

```text
N̂_Bs_200
```

上执行：

```text
Cosine K-Means
```

得到：

| 产物                  | 形状       | 用途               |
| ------------------- | -------- | ---------------- |
| C_k                 | (k, 200) | 聚类质心             |
| cluster_assignments | (N,)     | 每个名字属于哪个 cluster |

其中：

```text
k ≈ √N
```

---

## 1.4 构建列式矩阵

根据 cluster_assignments：

把 EL=50 的签名按 cluster 分组。

构建：

```text
M.shape = (k, max_size, 50)
```

流程图中：

* `C` 表示质心矩阵
* `M` 表示名称矩阵

但规范文档里：

| 流程图 | 规范文档 |
| --- | ---- |
| C   | C_k  |
| M   | C    |

建议统一：

```text
C_k = centroids
M = cluster_matrix
```

避免混淆。

## 3.1 核心计算

Server B 执行：

```text

```

即：

```text
CT-PT DotProduct
```

* 与所有明文质心
* 做点积

```text

```

即：

```text
查询与 k 个聚类中心的相似度
```

---

## 3.2 返回给 A

B 返回：

```text

```

# 七、Compute Sim（列式匹配核心阶段）

对应流程图：

```text
6/7/8. Compute Sim
```

这是协议核心。

---

# 7.1 Step 6：密文选择 cluster 内名字

B 对：

```text
column_j = M[:, j, :]
```

执行：

```text
E(selected_name_j)
= E(S) × column_j
```

由于：

```text
S 是 one-hot
```

因此等价于：

```text
在密文中选择：
最匹配 cluster 的第 j 个名字
```

但：

```text
B 不知道选的是哪个 cluster
```

---

# 7.2 Step 7：计算余弦相似度

执行：

```text
E(cos_score_j)
=
DotProduct_CT_CT(
    E(N̂_A_50),
    E(selected_name_j)
)
```

即：

```text
密文 × 密文
```

因为：

```text
两个向量都做过 L2 normalization
```

所以：

```text
dot product = cosine similarity
```

---

# 7.3 Step 8：减阈值并随机混淆

执行：

```text
E(score_j)
=
r_j × (E(cos_score_j) - τ)
```

其中：

```text
τ = 0.9
```

意义：

| 条件              | 结果          |
| --------------- | ----------- |
| cos_score_j > τ | score_j > 0 |
| cos_score_j < τ | score_j < 0 |

因此：

A 只需看：

```text
正负号
```

即可知道是否匹配。

但：

```text
乘随机数 r_j
```

后：

```text
A 无法反推出真实相似度
```

---

# 7.4 Early Stopping

流程图中特别标注：

```text
优化建议：
使用 early-stop 机制
```

即：

* B 可以逐列发送
* A 可以逐列解密
* 一旦发现：

```text
score_j > 0
```

即可提前结束。

无需等待所有列。

整个系统本质上是：

```text
MinHash 编码
+
K-Means 聚类缩小搜索空间
+
CKKS 同态加密实现隐私保护
+
列式矩阵实现密文候选选择
```

协议流程：

```text
B 离线聚类
→ A 加密查询
→ B 密文质心匹配
→ A 生成 one-hot
→ B 密文列式匹配
→ A 解密并判断
```

其中：

* CT-PT 用于降低计算成本
* CT-CT 仅用于最终余弦相似度
* Early-stop 用于降低延迟
* Cluster Matrix 用于避免线性扫描整个数据库
