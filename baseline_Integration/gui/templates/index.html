Step 1 — Database Clustering

Party B · offline preprocessing · runs once

Party B (the database holder) owns a database of real names. Each name is converted into a 200-dimensional MinHash signature, then cosine K-Means groups all signatures into K clusters.

Like a library shelving books by topic: similar names land in the same cluster. A later query only searches the most relevant cluster instead of scanning everything.

Privacy: this step runs entirely on B's machine — no communication, no data leaves B.

Step 4 — Cluster Selection

Party A · online · select best cluster after decryption

Party A (the requester) encrypts its query and sends it to B. B computes the query-to-centroid similarity under encryption and returns K encrypted scores.

A decrypts those scores with its private key. Each row is one query; columns are the K clusters and the colour is cosine similarity. The ★ brightest cell marks the most similar cluster, which A selects for exact matching.

Privacy: B only ever sees an encrypted query and encrypted scores — it never learns which cluster A picked. A learns its chosen cluster but not the names inside it.

Step 9 — Column-wise Match Detection

Party A · final decision · decrypt column scores

A sends B an encrypted selector (the chosen cluster) and its encrypted query vector. B multiplies the selector with the cluster matrix, scores each column under encryption, adds a random mask per column, and returns the ciphertext.

A decrypts column by column: a positive score means that name beats threshold τ. The first positive (★) is declared the hit. If no column is positive, the query is a miss.

Privacy: the per-column random mask prevents B from learning the true similarity. A sees scores only for names inside its chosen cluster — never the whole database.
